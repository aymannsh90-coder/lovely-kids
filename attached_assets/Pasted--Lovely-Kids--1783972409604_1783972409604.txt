أصلح الآن نظام استعادة كلمة المرور فقط في Lovely Kids.

الوضع الحالي:
- تسجيل الحساب والخروج والدخول مرة أخرى نجح من Preview.
- تسجيل الدخول يعتمد على رقم الجوال وكلمة المرور.
- البريد الإلكتروني محفوظ داخل الحساب ومخصص للاستعادة.
- واجهة "نسيت كلمة المرور؟" تطلب رقم الجوال وتقول إنها سترسل رابطًا إلى البريد المسجل.
- عند الضغط لا يصل أي بريد.

لا تعدّل تسجيل الدخول الحالي.
لا تعدّل Push Notifications.
لا تعدّل مشروع الويب الجديد.
لا تعمل git push أو EAS Build الآن.

المطلوب أولًا تشخيص ثم تنفيذ نظام استعادة حقيقي:

1. افحص هل يوجد endpoint حالي للاستعادة، مثل:
   POST /api/auth/forgot-password
   POST /api/auth/reset-password

2. افحص الزر الحالي في:
   artifacts/lovely-kids/app/(tabs)/profile.tsx
   وحدد هل ينفذ طلب API حقيقي أم يعرض رسالة نجاح وهمية فقط.

3. استخدم رقم الجوال فقط للبحث عن الحساب:
   - normalizePhone قبل البحث
   - لا تكشف هل الحساب موجود أم لا برسالة مختلفة، لأسباب أمنية

4. إذا وُجد الحساب وله بريد:
   - أنشئ reset token عشوائيًا وآمنًا
   - خزّن hash للتوكن فقط في قاعدة البيانات
   - ضع وقت انتهاء، مثل 30 دقيقة
   - اجعل التوكن يستخدم مرة واحدة فقط

5. أنشئ جدولًا واضحًا مثل:
   password_reset_tokens
   ويحتوي:
   - id
   - user_id
   - token_hash
   - expires_at
   - used_at
   - created_at

6. أرسل رابط الاستعادة إلى البريد المسجل باستخدام Resend.
   افحص هل RESEND_API_KEY موجود في Secrets.
   لا تطبع المفتاح أو التوكن في logs.

7. رابط الاستعادة يجب أن يفتح صفحة حقيقية تسمح للمستخدم بإدخال كلمة مرور جديدة.
   إذا تطبيق Android لا يستطيع استقبال الرابط مباشرة حاليًا:
   - أنشئ صفحة reset-password على API أو الويب الحالي
   - الرابط يحتوي reset token فقط
   - بعد نجاح التغيير اعرض رسالة واضحة للمستخدم

8. عند تعيين كلمة المرور الجديدة:
   - تحقق من التوكن
   - تحقق أنه غير منتهي وغير مستخدم
   - استخدم bcrypt.hash مرة واحدة
   - حدّث password_hash
   - علّم التوكن used
   - احذف أو أبطل الجلسات القديمة للمستخدم

9. لا تعرض رسالة نجاح وهمية.
   لا تعرض "تم الإرسال" إلا بعد قبول خدمة البريد للرسالة.

10. اختبر فعليًا:
   - طلب استعادة برقم جوال صحيح
   - وصول البريد
   - فتح الرابط
   - تعيين كلمة مرور جديدة
   - فشل الدخول بالكلمة القديمة
   - نجاح الدخول بالكلمة الجديدة
   - رفض إعادة استخدام الرابط
   - رفض الرابط بعد انتهاء صلاحيته

11. شغّل typecheck للتطبيق والـAPI.

12. لا تعمل commit قبل نجاح الاختبار الحقيقي.

أعطني تقريرًا يتضمن:
- هل الواجهة القديمة كانت وهمية أم مرتبطة بـAPI
- endpoint المستخدم
- اسم جدول reset tokens
- هل RESEND_API_KEY موجود
- البريد المرسل منه
- نتيجة إرسال Resend
- نتيجة اختبار تغيير كلمة المرور
- الملفات المعدلة
- نتيجة typecheck
