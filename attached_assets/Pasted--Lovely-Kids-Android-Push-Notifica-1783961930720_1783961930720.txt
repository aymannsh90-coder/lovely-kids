ابدأ الآن بإصلاح مشكلة تسجيل الدخول فقط في تطبيق Lovely Kids Android.

لا تعدّل نظام Push Notifications في هذه المرحلة.
لا تعدّل مشروع الويب.
لا تعمل git push أو EAS Build.

المعلومات التي ظهرت من التشخيص:
- تسجيل الدخول الأساسي لا يستخدم Clerk.
- يستخدم POST /api/auth/login.
- المصادقة مخصصة عبر bcrypt وجدول sessions.
- الخطأ الحالي من API هو HTTP 401:
  {"error":"البريد الإلكتروني أو كلمة المرور غير صحيحة"}
- التسجيل يسمح بإنشاء الحساب، لكن بعد logout يفشل الدخول بنفس البيانات.

المطلوب:

1. افحص كود التسجيل وكود تسجيل الدخول معًا لتحديد القيم الفعلية التي تُحفظ وتُستخدم:
   - email
   - phone
   - password hash
   - identifier

2. لا تفترض أن keyboardType هو السبب.
   أضف logging آمنًا مؤقتًا على الخادم يوضح:
   - هل identifier صُنّف كبريد أم هاتف
   - القيمة بعد normalization مع إخفاء معظمها
   - هل تم العثور على المستخدم
   - هل password verification نجح أو فشل
   لا تطبع كلمة المرور أو password hash أو session token.

3. افحص سجل المستخدم الذي تم إنشاؤه حديثًا في قاعدة البيانات:
   - email الفعلي المخزن
   - phone الفعلي المخزن
   - هل password_hash موجود
   - هل المستخدم مكرر
   - هل التسجيل أنشأه عبر Clerk فقط أم عبر جدول users أيضًا

4. طبّق دالة normalization موحدة مشتركة بين register وlogin:
   - trim للبريد وتحويله إلى lowercase
   - إزالة المسافات والشرطات والأقواس من الهاتف
   - توحيد الأرقام العربية إلى الإنجليزية إن لزم
   - توحيد صيغة أرقام فلسطين دون تغيير خاطئ للأرقام الدولية
   - لا تزل علامة + بشكل أعمى؛ حوّل الرقم إلى صيغة معيارية واضحة

5. اجعل login يبحث بطريقة صحيحة:
   - إذا identifier بريد: البحث بالبريد المطبع
   - إذا identifier هاتف: البحث بالهاتف المطبع
   - لا تستخدم مقارنة خام مختلفة عن التسجيل

6. تحقق من password hashing:
   - register يستخدم bcrypt.hash
   - login يستخدم bcrypt.compare مع نفس الحقل
   - لا يحدث double hashing
   - logout لا يعدل password أو user record

7. عدّل حقل identifier في الواجهة ليستخدم:
   keyboardType="default"
   autoCapitalize="none"
   autoCorrect={false}
   مع trim قبل الإرسال.

8. أزل useClerk/useSignIn من profile.tsx فقط إذا كانت غير مستخدمة في تدفق الدخول الأساسي.
   لا تضف ClerkProvider جديدًا لمجرد إصلاح هذا التدفق ما دام النظام الأساسي custom auth.
   وضح لماذا كانت Clerk hooks موجودة وما الوظيفة التي تستخدمها.

9. حسّن استجابة API ورسالة التطبيق بحيث نفرق بين:
   - المعرّف أو كلمة المرور غير صحيحة
   - بيانات ناقصة
   - خطأ اتصال
   - خطأ خادم

10. اختبر فعليًا عبر API:
   أ. أنشئ مستخدم اختبار مؤقتًا.
   ب. سجل الدخول مباشرة.
   ج. سجل الخروج/احذف الجلسة.
   د. سجل الدخول مرة ثانية بنفس البيانات.
   هـ. جرّب الهاتف بصيغ تحتوي مسافات وشرطات.
   و. جرّب البريد بحروف كبيرة وصغيرة.
   ز. احذف مستخدم الاختبار بعد انتهاء الاختبار إن أمكن بأمان.

11. شغّل typecheck والاختبارات ذات الصلة.

12. اعمل commit محلي فقط بعد نجاح الاختبار.

أعطني تقريرًا يتضمن:
- السبب الجذري المثبت، لا المتوقع.
- القيمة المخزنة مقابل القيمة المرسلة بعد إخفائها.
- الملفات المعدلة.
- نتيجة register → logout → login.
- نتائج اختبار الهاتف والبريد.
- هل Clerk hooks كانت مؤثرة أم لا.
- نتيجة typecheck.
